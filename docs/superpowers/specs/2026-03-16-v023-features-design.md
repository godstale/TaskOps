# TaskOps v0.2.3 Feature Changes

## Overview

Four changes for the v0.2.3 release:

1. Git branch renamed to `release/v0.2.3`
2. Remove `TASK_OPERATIONS.md` auto-generation
3. Add `plan update` command for syncing project plan changes to DB
4. Extend operation metadata: tool/skill/MCP usage, retry count, token usage

---

## Change 1: Branch Rename

Rename current branch to `release/v0.2.3`. Also update `__version__` in `cli/taskops.py` from `"0.3.0"` to `"0.2.3"`.

---

## Change 2: Remove TASK_OPERATIONS.md

### Rationale

`TASK_OPERATIONS.md` is auto-generated from the `operations` table. Since AI Agents can query the DB directly via `op log`, maintaining a redundant markdown file adds unnecessary overhead. Removing it reduces noise in project directories.

### Scope of Removal

- `cli/commands/query.py`: Remove `generate-ops` subparser and `handle_generate_ops()` function
- `cli/templates/TASK_OPERATIONS.md.tmpl`: Delete file
- `hooks/on_task_complete.sh`: No change needed (already does not call `generate-ops`)
- `skills/taskops.md`, `skills/taskops-gemini.md`, `SKILL.md`: Remove all references to `TASK_OPERATIONS.md`
- `docs/`: Update any documentation mentioning `TASK_OPERATIONS.md`
- `CHANGELOG.md`: Update v0.1.0 entry to remove the auto-generation mention

## Change 3: `plan update` Command

### Rationale

When a project plan changes, the AI Agent must apply those changes to the TaskOps DB to keep it in sync. A single `plan update` command accepts a JSON diff and applies it atomically, then regenerates `TODO.md`.

### Command Interface

```bash
python -m cli plan update --changes '<json-string>'
python -m cli plan update --changes-file ./path/to/changes.json
```

### JSON Schema

```json
{
  "create": [
    {
      "type": "epic | task",
      "title": "string (required)",
      "parent_id": "string (required for task; must reference existing epic or task)",
      "description": "string (optional)",
      "status": "todo | in_progress | interrupted | done | cancelled (optional, default: todo)"
    }
  ],
  "update": [
    {
      "id": "string (required)",
      "title": "string (optional)",
      "description": "string (optional)",
      "status": "todo | in_progress | interrupted | done | cancelled (optional)"
    }
  ],
  "delete": [
    {
      "id": "string (required)"
    }
  ]
}
```

- Any of `create`, `update`, `delete` may be omitted or empty. An empty `{}` is valid (no-op, succeeds).
- `objectives` are not managed by `plan update`; use the `objective` command directly.
- User-provided IDs in `create` items are ignored; IDs are always auto-generated by the existing sequence logic.

### Processing Rules

**Atomicity model:**
1. Parse and validate JSON (fail fast with clear error if JSON is malformed).
2. Validate all items before touching the DB:
   - For `create`: verify `parent_id` exists (if provided).
   - For `update`: unknown IDs print a warning and are collected; valid IDs proceed.
   - For `delete`: unknown IDs print a warning and are collected; valid IDs proceed.
3. If any `create` item has an invalid `parent_id`, that item is skipped with a warning — it does **not** abort the entire batch.
4. Apply all valid operations in a single DB transaction. On any DB error, the entire transaction rolls back.
5. After a successful commit, regenerate `TODO.md`. If `TODO.md` regeneration fails, print a warning but do **not** roll back the DB changes (audit log takes priority).

**Delete behavior (plan update only):** Cascading delete — deletes the target task and all its children recursively. Operations linked to deleted tasks are preserved for audit. This is distinct from `task delete`, which refuses to delete if children exist.

### New File

`cli/commands/plan.py` — new module following the same pattern as other command modules.

### Registration in `cli/taskops.py`

```python
from .commands.plan import register as register_plan
# in build_parser():
register_plan(subparsers)
```

### Skill Document Update

Add a section to `skills/taskops.md` and `skills/taskops-gemini.md`:

> **Plan Changes:** When the user modifies a project plan (adds, removes, or renames tasks or epics), call `plan update` with a JSON description of the changes before continuing execution. Example:
> ```bash
> python -m cli --db taskops.db plan update --changes '{"update":[{"id":"PRJ-T001","title":"New title"}],"create":[{"type":"task","parent_id":"PRJ-E001","title":"New task"}]}'
> ```

## Change 4: Extended Operation Metadata

### Rationale

Each operation log entry should capture which tools, skills, and MCP servers were used, how many retries occurred, and token consumption. The Agent provides this information explicitly when logging operations, for TaskBoard visualization.

### Schema Changes

New columns added to the `operations` table (schema version 1 → 2):

| Column | Type | Default | Description |
|---|---|---|---|
| `tool_name` | TEXT | NULL | Tool used (e.g., Edit, Write, Bash, Agent) — free text |
| `skill_name` | TEXT | NULL | Skill invoked — free text |
| `mcp_name` | TEXT | NULL | MCP server used — free text |
| `retry_count` | INTEGER | 0 | Number of retries; must be >= 0 |
| `input_tokens` | INTEGER | NULL | Input tokens consumed; must be >= 0 if provided |
| `output_tokens` | INTEGER | NULL | Output tokens consumed; must be >= 0 if provided |
| `duration_seconds` | INTEGER | NULL | Elapsed wall-clock time in seconds; must be >= 0 if provided |

All new fields are optional. Tool/skill/MCP names are free text (no enum restriction).

### Migration

**`cli/db/schema.py`:**

- `SCHEMA_VERSION` bumped from `1` to `2`
- `SQL_CREATE_OPERATIONS` updated with new columns (for fresh installs)
- `SQL_MIGRATE_V1_TO_V2`: sequence of `ALTER TABLE` statements:
  ```sql
  ALTER TABLE operations ADD COLUMN tool_name TEXT;
  ALTER TABLE operations ADD COLUMN skill_name TEXT;
  ALTER TABLE operations ADD COLUMN mcp_name TEXT;
  ALTER TABLE operations ADD COLUMN retry_count INTEGER DEFAULT 0;
  ALTER TABLE operations ADD COLUMN input_tokens INTEGER;
  ALTER TABLE operations ADD COLUMN output_tokens INTEGER;
  ALTER TABLE operations ADD COLUMN duration_seconds INTEGER;
  ```
- `migrate_schema(conn)` function:
  1. Read current version from `settings` table (`key = '__schema_version'`); default to `1` if not found.
  2. If current version < 2: run `SQL_MIGRATE_V1_TO_V2` statements inside a transaction, then upsert `__schema_version = 2` into `settings`.
  3. Idempotent: if version already >= 2, do nothing.
  4. On failure, the transaction rolls back and the error propagates to the caller.

**`cli/db/connection.py`:** After `create_tables(conn)`, call `migrate_schema(conn)`.

### CLI Changes

**`op progress`** — add options (all optional):
```
--tool TEXT        Tool used in this step (e.g., Edit, Write, Bash, Agent)
--skill TEXT       Skill invoked
--mcp TEXT         MCP server used
```

**`op complete`** — add options (all optional):
```
--tokens-in INT    Input tokens consumed (>= 0)
--tokens-out INT   Output tokens consumed (>= 0)
--retry-count INT  Number of retries, default 0 (>= 0)
--duration INT     Duration in seconds (>= 0)
```

**`op start`** — no changes.

**`op log`** — display new fields when present (show `tool_name`, `skill_name`, `mcp_name`, `retry_count`, `input_tokens`, `output_tokens`, `duration_seconds` for each row if non-null/non-zero).

### Skill Document Update

Add to `skills/taskops.md` and `skills/taskops-gemini.md` in the operation logging section:

> When logging `op progress`, pass `--tool <tool-name>` for each significant tool use, and `--skill` or `--mcp` if a skill or MCP server was invoked.
> When logging `op complete`, include `--tokens-in`, `--tokens-out`, `--retry-count`, and `--duration` if known.
